# Workbench Mission — 目的・対象・成功条件の正本

**文書の目的**: このドキュメントは、pbg-filemaker-migration-workbench の存在理由・解くべき問題・成功条件・運用前提を定義します。実装判断・機能優先度・設計方針のすべてはこの文書を根拠とします。

---

## このシステムは何か

**稼働中の FileMaker を止めずに、データの意味を現場と一緒に解明し、差分に耐える形へ整備するワークベンチ。**

具体的には:
- FileMaker から出力した CSV / XLSX を受け取り、「これは何のファイルか」「この列は何か」を確認・記録する
- ファイル種別 / 列の意味 / 入力規則 / 要不要 / 現場独自語を分類・既知化していく
- 一度確認した設定（profile / mapping）を積み上げて、次回の確認コストを下げる
- 巨大件数（アポリスト約 283 万件）はテンプレート適用 + 差分確認中心で処理する

## このシステムは何ではないか

| よくある誤解 | 正しい理解 |
|------------|-----------|
| 一回実行して終わる移行スクリプト | 繰り返し読み込み・差分チェック・再投入が前提 |
| 完成した本番業務システム | Ops Core v1 の前段。暫定的な整備ツール |
| 全件人手レビューを前提とするシステム | システムが候補を出し、人は意味づけと例外確認をする |
| FileMaker の代替 / 後継システム | FileMaker は稼働し続ける |
| 汎用 ETL / BI ツール | pbg 固有の FileMaker データ整備に特化 |
| 自動移行・自動確定システム | 候補・提案は出すが、確定は人が行う |

---

## 解くべき問題

現在の FileMaker には以下の課題がある:

1. **ファイル種別が分からない** — 74 種以上のファイルが何のデータか文書化されていない
2. **列の意味が分からない** — フィールド名が略語・社内造語で、何が入っているか不明
3. **ヘッダーなし CSV が多い** — 列が位置で定義されており、名前がない
4. **入力規則が不明** — 何を入れてよいか、必須か任意かが定義されていない
5. **稼働を止められない** — 太陽光・エアコンなどのファイルは平日休日を問わず読み書きされる
6. **一回では終わらない** — データは日々更新されるため、差分再投入・再統合が繰り返し発生する

---

## 対象データ

- FileMaker から出力した CSV（Shift-JIS / cp932 が多い）
- FileMaker から出力した XLSX（大容量は CSV 変換を推奨）
- ヘッダーあり / ヘッダーなし の両方
- アポリスト（約 283 万件）、顧客一覧、コール履歴、訪問履歴、その他業務ファイル群

## 非対象

- Ops Core v1 本番業務データの直接操作
- FileMaker への書き戻し
- 外部サービス / クラウドへのデータ送信
- FileMaker 以外の基幹系（ERP / CRM 等）との連携
- 最終的なデータ品質保証（保証できるのは「手順と記録」）

---

## 成功条件

以下が満たされたとき、このシステムは目的を果たしている:

1. **既知化の進行** — ファイル種別・列の意味・入力規則が profile / review として積み上がっている
2. **差分耐性の獲得** — 同じファイルを再度読み込んでも安全に処理できる
3. **確認コストの低減** — 2回目以降の同種ファイルは「このまま進む」で処理できる
4. **現場が参加できる** — 専門知識なしでも、提示された選択肢に答えられる UI がある
5. **原本が保全されている** — いつでも最初から再実行できる状態が維持されている

**成功の指標ではないもの**: 処理速度、100% 自動化率、全件人手確認完了

---

## 運用前提

- **FileMaker は稼働し続ける** — このシステムは FileMaker に干渉しない。読み取り専用で CSV / XLSX を受け取るのみ
- **ローカル実行のみ** — クラウド送信・外部 API 接続なし
- **繰り返し実行が前提** — 同じファイルの2回目、3回目の読み込みは日常業務
- **現場担当者が使う** — 高い IT リテラシーを前提としない UI が必要
- **記録を残す** — 誰が・いつ・何を確認したかが後から分かるようにする

---

## アポリストの扱い

- 件数: 約 **283 万件**
- 人が全件レビューする前提は捨てる
- テンプレート（profile）を適用し、差分・異常値・未分類のみを人が確認する
- sourceBatchId / schemaFingerprint による差分追跡を活用する
- 重複候補は「候補」のまま出力し、自動マージはしない

---

## 稼働中ファイルの扱い

- 太陽光・エアコンなど主要ファイルは **平日・休日を問わず読み書きされている**
- CSV / XLSX は FileMaker から「エクスポート」されたスナップショットとして受け取る
- このシステムは FileMaker に書き込まない
- スナップショット時点のデータを処理するため、再実行時は最新のエクスポートを使う

---

## 差分再投入 / 再統合前提

- 一回で終わる前提を捨てる
- 新しいエクスポートが来るたびに、前回との差分を検出・確認・再投入する
- `sourceBatchId`（ファイル内容のハッシュ）と `schemaFingerprint`（スキーマハッシュ）で同一性を追跡する
- `previousRunId` で前回 run との連続性を保つ

---

## 人の役割とシステムの役割

システムは候補・提案・分類を自動化するが、**確定は人が行う**。

| 局面 | システムがすること | 人がすること |
|------|-----------------|------------|
| ファイル種別確認 | candidate を confidence ラベル付きで提示 | 「これで合っている」「別の種別にする」を選ぶ |
| 列の意味確認 | profile の列定義を提示し、「わからない」も選択肢に含める | 意味・要不要・入力規則を入力 |
| 2回目以降の読み込み | fast path で「このまま進む」を提示 | 確認が必要なら「列を確認する」を選ぶ |
| 差分検出 | 前回との差分を出力 | どの差分を取り込むか判断 |
| 重複検出 | 候補を出力 | どれが本物の重複か判断 |

現場回答は **candidate / proposal** であり、**canonical truth ではない**。
回答内容は profile や effective mapping として保存され、次回以降の候補精度を上げる材料になる。

---

## known file / candidate profile / effective mapping / review の位置づけ

| 概念 | 役割 | 変更のライフサイクル |
|------|------|-----------------|
| seed profile (built-in) | 既知ファイル種別の雛形。provisional=true | システム更新時のみ変更 |
| candidate profile | 現場レビューから生成された「仮の設定」。重ねて使う | run ごとに積み上げ |
| column review | 現場が「この列は何か」を回答したデータ | run-scoped。profile 本体は変えない |
| effective mapping | review 回答から生成した run 単位の列 mapping | run ごとに独立 |
| fast path | high confidence の known file で「このまま進む」を提供 | 便利機能。unsafe な自動確定はしない |

---

## Ops Core v1 との関係

- このシステムは Ops Core v1 の **前段整備ツール**
- 目的: FileMaker データの意味を解明し、Ops Core v1 に渡せる状態に整備すること
- Ops Core v1 の設計・仕様はこのシステムが決めない
- このシステムが終わったら Ops Core v1 が動き始める、ではなく、**両者は並走する期間がある**（未確定）

---

## 「最終移行」ではなく「整備・既知化・差分耐性の獲得」が目的

このシステムの完了条件は「全データを移した」ではなく:

- FileMaker の主要ファイル群が既知化されていること
- 繰り返し差分処理が安定して動いていること
- 現場担当者が自力で確認作業を進められること
- 設定（profile / mapping）が積み上がり、次の処理が楽になっていること

**移行完了ではなく、整備基盤の確立が成功定義。**
